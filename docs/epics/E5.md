# E5 - X + transcripts (M5) - brief: twscrape ops runbook (from xfilter), transcript caps

> ~1 page, written once for the epic. Its job: let an agent implement any E5
> issue without reading other epics' code. PRD §5-§9, §12-§13 are the source of
> truth; this restates the interfaces E5 consumes (E1's `Fetcher`) and produces
> (the X fetcher, the transcript store, the transcript excerpter).

## Goal

Add the two source classes the PRD deliberately deferred past daily-usable
(M4): **X accounts** watched via twscrape (GRP-50/51) and **YouTube
transcripts** feeding keyword extraction (GRP-52/53). Both are **best-effort**
(PRD §13 risk table): the site is fully functional without either, so every
failure mode degrades to skip + log and never fails the run. Nothing here
changes an architecture decision (PRD §5) - X plugs into the existing
`Fetcher` contract unchanged, transcripts ride the existing `transcript_ref`
column, and excerpts ride the existing extraction batcher.

## Interfaces consumed (restated from E1 - `docs/epics/E1.md`)

### `RawItem` + `Fetcher` (`grepify.ingest`) - GRP-10

A fetcher emits one `RawItem` per feed entry *before* identity/hashing; the
normalizer (GRP-14) computes `item_id`/`content_hash`/`canonical_url`. The X
fetcher implements the identical contract:

```python
class RawItem(BaseModel):          # frozen, extra=forbid
    external_id: str | None = None # tweet id
    url: str                       # tweet permalink
    title: str                     # tweet text (X has no title column - see below)
    summary: str | None = None
    author: str | None = None
    published_at: str | None = None
    lang: str | None = None
    transcript_ref: str | None = None  # youtube only

class Fetcher(ABC):
    @property
    def kind(self) -> SourceKind: ...
    def fetch(self, source: Source) -> list[RawItem]: ...
```

`Fetcher.fetch` contract that E5 must honor (the failure-isolation design
GRP-15/16 depends on):

- An **empty** result is `return []`, not an error.
- Any **per-source** failure MUST be raised as `FetchError` (`grepify.errors`).
  The orchestrator (GRP-15) catches it, writes an `error` `fetch_log` row, and
  continues the run. One dead source never fails the run (PRD §9). This is the
  isolation contract E5's "X is best-effort" rule (PRD §13) reduces to: every X
  failure mode (login challenge, rate limit, suspended account, no accounts
  configured) surfaces as `FetchError`, never anything the orchestrator does
  not already isolate. `FakeTweetSource` + fixtures let a test assert this
  without a network.
- Per-run per-source caps (F-ING-06, default 50) are applied by the
  fetcher/orchestrator, not baked into the contract.

`FetcherRegistry` maps one fetcher per `SourceKind`; the X fetcher is one
`register(XFetcher(...))` call in `build_registry`, no orchestrator change.

### `SettingsConfig.limits` (`grepify.config.schemas`)

`transcript_max_chars` (default 60000) and `transcript_langs` (default `[en]`)
are the storage cap and language preference the transcript store reads (PRD §7).

## Interfaces produced (this epic)

### X fetcher (`grepify.ingest.x`) - GRP-50

```python
class Tweet(BaseModel):            # frozen, extra=forbid - the twscrape-neutral value type
    id: str                        # tweet id (snowflake, as string)
    url: str
    text: str
    author: str | None = None
    created_at: str | None = None  # ISO-8601
    lang: str | None = None

class TweetSource(Protocol):       # the injectable seam (like ingest.http.Transport)
    def tweets(self, handle: str, *, since_id: str | None, limit: int) -> list[Tweet]: ...

class XFetcher(Fetcher):           # kind == SourceKind.X
    def __init__(self, source: TweetSource | None, *,
                 since_ids: SinceIdProvider = ..., limit: int = 50) -> None: ...
```

- **Tweet text -> `RawItem.title`.** The PRD §6 `items` schema has no tweet-text
  or metrics column (same as reddit `score`, which E1 drops); the tweet text is
  the only field carrying signal, so it becomes `title` (collapsed to one line)
  so keyword extraction (E2) sees it unchanged and the site shows it. Metrics
  are read and deliberately dropped - there is nowhere in the contract to put
  them.
- **since_id tracking (F-ING-05).** `SinceIdProvider = Callable[[str], str | None]`
  maps a `source_id` to the highest tweet id already stored for it, so twscrape
  is asked only for tweets newer than the last seen one. It is **derived from
  truth** (the max `external_id` among that source's stored items), not a
  separate committed state file: CI containers are ephemeral (PRD §5), truth on
  the `data` branch is the one durable store, and deriving keeps since_id from
  ever drifting from what was actually written. The default provider returns
  `None` (fetch recent, let `item_id` dedup handle idempotency - still correct,
  just less efficient); the `ingest` CLI wires the real one from the repository.
- **Unconfigured is a skip, not a crash.** `XFetcher(None, ...)` (no accounts /
  twscrape not installed) raises `FetchError` from `fetch`, so an X source with
  no live wiring degrades to a logged skip exactly like a rate limit - it is
  registered regardless so an enabled X source never hits the systemic
  "no fetcher for kind" `KeyError`.

### twscrape account/session management (`grepify.ingest.x_twscrape`) - GRP-50

`TwscrapeTweetSource` is the real `TweetSource`, importing twscrape lazily
(optional `x` extra). Accounts come from CI secrets, never the repo (PRD §5):

- **Secret `GREPIFY_X_ACCOUNTS`** - a JSON array of burner accounts, each
  `{"username","password","email","email_password"}` (twscrape's
  `add_account` argument shape; a `cookies` string may replace the password
  pair for a pre-authed session). Read once, added to twscrape's account pool,
  `login_all()` refreshes sessions. **`GREPIFY_X_ACCOUNTS_DB`** (optional) is
  the path to twscrape's accounts SQLite db so a cached, logged-in session
  survives between runs (the CI cache); absent, accounts re-auth each run.
- **Failure modes -> `FetchError` (skip + log), never past the orchestrator:**
  a login **challenge**, a **rate limit** (twscrape exhausts its pool), a
  **suspended/locked** account, or an empty account pool. `TwscrapeTweetSource`
  maps each twscrape error class (and asyncio/`RuntimeError` transport faults)
  onto `FetchError` with a diagnostic message; nothing twscrape raises escapes
  as anything else. This is the "X is best-effort" isolation contract (PRD §13)
  and is tested with a fake that raises each class.
- **Never logged:** credentials, cookies, session headers (PRD §5 security).

`FakeTweetSource` (shipped in `grepify.ingest.x`, not `tests/`) is the
fixture-based double every X test uses: canned `Tweet`s per handle, a canned
exception to raise, and a recorded `.calls` list of `(handle, since_id)`.

### Transcript store (`grepify.ingest.transcript`) - GRP-52

```python
class TranscriptClient(Protocol): # injectable seam; real impl wraps youtube-transcript-api lazily
    def fetch(self, video_id: str, *, languages: Sequence[str]) -> str | None: ...

class TranscriptStore:
    def ensure(self, video_id: str) -> str | None: ...  # -> transcript_ref (relative path) or None
    def read(self, transcript_ref: str) -> str | None: ...
```

- **Absence is OK (F-ING-03).** No transcript (disabled, none published,
  blocked from CI IPs - PRD §13) -> `ensure` returns `None`, the item stores
  `transcript_ref=null`. **Every** client failure degrades to `None`; the store
  never raises for a missing/unfetchable transcript.
- **Compression + cap.** Text is truncated to `transcript_max_chars` (PRD §7
  limits) then gzip-compressed under `data/transcripts/<video_id>.txt.gz`.
- **Idempotent (F-ING-07).** If the blob already exists, `ensure` returns its
  ref without calling the client - transcripts already stored are never
  re-fetched. The `YouTubeFetcher` takes an optional store and, when present,
  sets `RawItem.transcript_ref` per video; when absent, behavior is unchanged.

### Transcript excerpting (`grepify.ingest.transcript`) - GRP-53

```python
def excerpt_transcript(text: str, *, max_chars: int = 1500) -> str: ...
```

First `max_chars` with a **smart cut** back to the last sentence/word boundary
(F-EXT-01: "transcript-excerpt <= 1500 chars for youtube"). The extract prompt
(GRP-21) gains an optional `transcript_reader` so a youtube item's excerpt is
appended to its prompt payload; non-youtube items and items without a transcript
are unchanged. The eval harness (GRP-24) re-scores the prompt on any change
(PRD §10.5) - the eval delta is reported in the MR.
