# E2 - Extraction (M2) - brief: LLM provider contract, budget rules, fallback

> ~1 page, written once for the epic. Its job: let an agent implement any E2
> issue (and the E4 digest generator, which reuses the LLM provider) without
> reading other epics' code. PRD §5-§10, §12 are the source of truth; this
> restates the interfaces E2 produces and consumes.

## Goal

Turn untagged items into keyword rows, cheaply and safely. Every new item gets
≥1 keyword - from a small/cheap LLM when it is available and in budget, from a
deterministic local extractor (GRP-22) otherwise - so the trend surface (E3/E4)
always has data and the site always builds (PRD §5 "deterministic fallback",
§9 "LLM failing never blocks site build"). The load-bearing pieces built first
are the **LLM provider** (GRP-20) - the only code that touches a model, wrapping
it in bounded retries and a **hard per-run budget circuit breaker** (the CSR
retry-loop lesson, PRD §5) - and the **extract batcher + prompt v1** (GRP-21),
which is the strict-JSON contract, response validation, and per-batch fallback.
GRP-22 (YAKE fallback), GRP-23 (normalize/alias/mute), GRP-24 (eval), and GRP-25
(pipeline wiring + data-quality gates) plug into these.

## Interfaces consumed (restated)

### `ConfigProvider` (`grepify.config`)

```python
def settings(self) -> SettingsConfig: ...   # .llm.active_profile, .llm.profiles, .llm.max_items_per_call
```

`SettingsConfig.llm` is an `LlmSettings` (`grepify.config.schemas`):

```python
class LlmSettings:  active_profile: str; max_items_per_call: int = 25; profiles: dict[str, LlmProfile]
class LlmProfile:   endpoint: str; model: str | None; cmd: str | None; max_calls_per_run: int | None
```

`endpoint` is a profile *type* marker (`openai-compat` | `anthropic` | `vertex`
| `cli`), not a URL. The concrete endpoint base URL and API key are **deployment
secrets** (PRD §5 "LLM key … in CI secrets; never in repo"), resolved at wiring
time (GRP-25) from the environment (`LLM_BASE_URL` / `LLM_API_KEY`), never from
committed config. The client takes them as explicit constructor params so tests
inject them directly and no secret is ever read or logged inside the module.

### `Repository` (`grepify.repository`)

```python
def log_llm(self, entry: LlmLogEntry) -> None: ...          # one row per real call (GRP-20)
def add_item_keywords(self, keywords: Sequence[ItemKeyword]) -> int: ...  # idempotent (GRP-25)
def existing_item_ids(self) -> set[str]: ...                # cache/untagged selection (GRP-25)
```

The provider depends only on the narrow log sink `Callable[[LlmLogEntry], None]`,
which `Repository.log_llm` satisfies - no backend type leaks (Postgres-swappable).

### Domain models (`grepify.models`)

`Item` (title/summary are the extraction input; PRD §6 column set), `ItemKeyword`
(item_id, keyword, rank, method ∈ {`llm`,`fallback`}, model, extracted_at),
`LlmLogEntry` (run_id, purpose ∈ {`extract`,`digest`}, model, input_items,
tokens_in, tokens_out, status, created_at). `Clock` (`grepify.clock`) supplies
timestamps; the module never calls `datetime.now()` directly (PRD §5).

## Interfaces produced (later E2/E4 issues consume these)

### LLM provider (`grepify.llm`) - GRP-20

```python
class ChatMessage(role: str, content: str)                  # frozen
class Completion(text: str, tokens_in: int | None, tokens_out: int | None)  # frozen

class LlmClient:
    calls_made: int                                          # budget usage (read-only)
    max_calls_per_run: int | None
    def complete(self, messages: Sequence[ChatMessage], *,
                 run_id: str, purpose: str, input_items: int) -> Completion: ...

def build_client(profile, *, api_key, base_url, log_sink, clock, **opts) -> LlmClient: ...
```

`complete` contract:

- **Budget breaker (per profile).** One `complete()` = one budget unit. Before
  any network I/O it checks `calls_made >= max_calls_per_run`; if so it raises
  `BudgetExceededError` **without touching the transport** and without incrementing -
  the AC test is that the 41st call at a cap of 40 is refused with zero network
  calls. `max_calls_per_run = None` means "no cap" (never used in v1, where every
  profile sets one). The breaker is a hard ceiling, never an unbounded loop.
- **Bounded, jittered retries.** A single `complete()` retries transient
  failures (transport error, HTTP 429/5xx) at most `max_attempts` times (default
  3) with exponential backoff + jitter (`sleep`/`rng` injected for determinism).
  A non-retryable 4xx short-circuits. Retries live *inside* one budget unit;
  exhausting them raises `LlmError`.
- **`llm_log` for every real call.** Exactly one `LlmLogEntry` is written per
  `complete()` that reaches the transport - `status='ok'` with token usage on
  success, `status='error'` when retries are exhausted. A `BudgetExceededError`
  refusal writes **no** row (it is not a call). Headers/keys are never logged
  (PRD §5 security).
- Only `endpoint='openai-compat'` is implemented this session; `build_client`
  raises `LlmError` for other endpoints (anthropic/vertex/cli are later work).

### Extract batcher + fallback hook (`grepify.extract`) - GRP-21

```python
class FallbackExtractor(Protocol):
    def extract(self, items: Sequence[Item]) -> dict[str, list[str]]: ...  # item_id -> keywords; GRP-22 impls

def run_extract(items, client, *, run_id, clock, fallback,
                max_items_per_call=25, max_keywords=8,
                max_malformed_retries=...) -> ExtractResult: ...

class ExtractResult:  keywords: list[ItemKeyword]; batches_total/llm/fallback: int
                      llm_retries: int; budget_exhausted: bool
```

`FallbackExtractor` is defined here (the consumer owns the interface); GRP-22
implements it with YAKE. `run_extract` never raises for LLM problems - it
degrades to fallback and always returns keyword rows.

## Contracts & rules

- **Batching (F-EXT-01).** Items are chunked to ≤ `max_items_per_call` (25 per
  `settings.yml`). Each batch is one prompt: title + summary (summary capped for
  token hygiene). Transcript excerpts (F-EXT-01, YouTube) are **E5/GRP-53**, not
  this epic.
- **Strict-JSON contract (prompt v1).** Model must return exactly
  `[{"item_id": "...", "keywords": ["...", …≤8]}]` and nothing else. Prompt
  version is a constant (`extract-v1`); the §6 `item_keywords` schema records
  provenance via `model` only (no `prompt_version` column - see PR note).
- **Validation (F-EXT-02), batch-level.** A response is valid iff: it parses as
  JSON; it is a list of `{item_id, keywords}` objects; the returned `item_id`
  set **equals** the batch's item_id set (echo check, no unknowns/dupes/misses);
  every keyword passes sanity (2-60 chars after trim, contains no URL) and each
  item has ≤ `max_keywords`. Any violation ⇒ the batch is malformed.
- **Retry-then-fallback (F-EXT-02).** A malformed batch is retried **once**
  (same prompt), then that batch falls back to the deterministic extractor.
  `LlmError` (network exhausted) and `BudgetExceededError` short-circuit straight to
  fallback for the affected batch; once budget is exhausted **all** remaining
  batches go to fallback without further LLM calls. Retries are **bounded in
  total** (`max_malformed_retries`) so a pathological run can't retry forever.
- **Keyword rows.** LLM keywords → `method='llm'`, `model=<profile model>`;
  fallback keywords → `method='fallback'`, `model=None`. `rank` = 1-based
  position in the returned/fallback order. Full normalization + alias/mute is
  **GRP-23**; this epic only trims and sanity-checks.
- **Cache (F-EXT-04).** Never re-extract an item that already has keywords
  unless `--force`; selecting untagged items and honoring `--force` is **GRP-25**
  wiring - the batcher extracts exactly the items it is handed.
- **Failure modes.** LLM down / over budget / malformed ⇒ degrade to fallback
  (run continues, site builds). Systemic faults (bad config, unreadable truth)
  propagate from the layers that own them; the LLM module adds none of its own
  process-failing paths.
