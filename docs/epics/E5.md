# E5 - Transcripts + long-form AI voices (M5) - brief: transcript caps, feed-over-X

> ~1 page, written once for the epic. Its job: let an agent implement any E5
> issue without reading other epics' code. PRD §5-§9, §12-§13 are the source of
> truth; this restates the interfaces E5 consumes (E1's `Fetcher`) and produces
> (the transcript store + the transcript excerpter).

## Goal

Deepen the content the pipeline extracts, without adding an always-on
integration or an auth-walled source:

- **YouTube transcripts (GRP-52/53)** - fetch, cache, cap, and excerpt a video's
  transcript so keyword extraction sees the talk, not just the title.
- **Long-form AI voices (replaces GRP-50/51)** - a plain RSS group of the
  individual practitioners' blogs/Substacks, `sources/groups/ai-voices.yml`.

Both are **best-effort** (PRD §13): the site is fully functional without either.
Nothing here changes an architecture decision (PRD §5) - transcripts ride the
existing `transcript_ref` column and the existing extraction batcher; the voices
group rides the existing RSS fetcher.

### Scope change from the PRD (Kyle-approved this session)

PRD §12 scoped GRP-50 (twscrape X fetcher + CI account/session management) and
GRP-51 (X items in site). **Both were dropped in favour of the AI-voices RSS
group.** Rationale: the watchlist people (Willison, Mollick, Lambert, swyx)
publish long-form on their own feeds, which is *richer* for keyword extraction
than 280-char tweets and reads over the RSS machinery that already works - no
burner accounts, no session cookies, no per-run login challenges (PRD §13 named
twscrape breakage as a High risk, and X login from CI IPs is the worst of it).
This is a **PRD §12 diff proposed in the MR** (CLAUDE.md: never change scope
silently); the PRD text itself is left for Kyle to amend. The GRP-50/51 issue
numbers are retired; GRP-52/53 are unchanged.

## Interfaces consumed (restated from E1 - `docs/epics/E1.md`)

### `RawItem` + `Fetcher` (`grepify.ingest`) - GRP-10

A fetcher emits one `RawItem` per feed entry *before* identity/hashing; the
normalizer (GRP-14) computes `item_id`/`content_hash`/`canonical_url`.
`RawItem.transcript_ref` (youtube only, else `None`) is the field E5 fills.

### `SettingsConfig.limits` (`grepify.config.schemas`)

`transcript_max_chars` (default 60000) and `transcript_langs` (default `[en]`)
are the storage cap and language preference the transcript store reads (PRD §7).

## Interfaces produced (this epic)

### Transcript store (`grepify.ingest.transcript`) - GRP-52

```python
class TranscriptClient(Protocol): # injectable seam; real impl wraps youtube-transcript-api lazily
    def fetch(self, video_id: str, *, languages: Sequence[str]) -> str | None: ...

class TranscriptStore:
    def ensure(self, video_id: str) -> str | None: ...  # -> transcript_ref (relative path) or None
    def read(self, transcript_ref: str) -> str | None: ...
```

- **Absence is OK (F-ING-03).** No transcript (disabled, none published, or
  blocked from CI IPs - PRD §13) -> `ensure` returns `None`, the item stores
  `transcript_ref=null`. **Every** client failure degrades to `None`; the store
  never raises for a missing/unfetchable transcript.
- **Compression + cap.** Text is truncated to `transcript_max_chars` (PRD §7)
  then gzip-compressed (fixed mtime = deterministic bytes) under
  `data/transcripts/<video_id>.txt.gz`; `transcript_ref` is that path relative
  to the data root (posix), portable and diff-stable.
- **Idempotent (F-ING-07).** If the blob already exists, `ensure` returns its
  ref without calling the client. `video_id` is validated (`[A-Za-z0-9_-]`)
  before it touches the filesystem; `read` rejects a ref that escapes the data
  root. The `YouTubeFetcher` takes an optional store and, when present, sets
  `RawItem.transcript_ref` per video (from the bare `yt:videoId`); when absent,
  behavior is unchanged.
- The real client (`YouTubeTranscriptApiClient`) imports youtube-transcript-api
  lazily (optional `transcripts` extra); tests use a fake, so this is never
  unit-tested against a network.

### Transcript excerpting (`grepify.ingest.transcript`) - GRP-53

```python
def excerpt_transcript(text: str, *, max_chars: int = 1500) -> str: ...
```

First `max_chars` with a **smart cut** back to the last sentence/word boundary
(F-EXT-01: "transcript-excerpt <= 1500 chars for youtube"). The extract prompt
(GRP-21) gains an optional `transcript_reader` so a youtube item's excerpt is
appended to its prompt payload; the augmentation is **additive and conditional**
- a batch with no transcript-bearing item gets the byte-identical v1 prompt, and
the output contract (`[{item_id, keywords}]`) is unchanged, so `PROMPT_VERSION`
stays `extract-v1`.

### AI-voices RSS group (`sources/groups/ai-voices.yml`) - GRP-51 replacement

A `category: ai` group of the watchlist practitioners' long-form feeds. Pure
config over the existing RSS fetcher; `grepify validate` pings each feed on the
MR, and a feed that cannot be confirmed live ships `enabled: false` (GRP-07
convention) rather than being omitted.
