# E4 - Digests + keyword pages (M4) - brief: digest data contract, url schemes, rising math

> ~1 page, written once for the epic. Its job: let an agent implement any E4
> issue without reading other epics' code. PRD §5-§10, §12 are the source of
> truth; this restates the interfaces E4 consumes (E2 LLM provider, E3 render +
> trends) and the contracts E4 produces. **Tag v1.0.0 ships here** (daily-usable).

## Goal

Turn the derived cache into per-category digests and per-keyword detail pages -
the two surfaces that make the site worth opening every morning (PRD §4 flow 1
and 3). Two design-critical pieces come first: **rising detection + the digest
input assembler** (GRP-40, deterministic, category-keyed) and **daily digest
generation** (GRP-41, one LLM call per category, degrades to a deterministic
template when the LLM is unavailable/over budget). Weekly (GRP-42) reuses that
machinery on an ISO-week window; digest pages (GRP-43) and keyword pages
(GRP-44) render it; cron gating (GRP-45) fires the digest steps in the right
America/Edmonton time-of-day window. Everything stays a **pure function of the
cache + config + injected clock** (F-SIT-08) except the single LLM call per
category, which is offline-faked in tests (no network, PRD §9/§10).

## Interfaces consumed (restated)

### Render skeleton + trend queries (`grepify.site`) - E3

```python
def create_environment() -> jinja2.Environment                     # deterministic Jinja env
def render_page(env, template, ctx, **data) -> str                 # bytes-stable html
@dataclass(frozen=True) class SiteMeta:  title; base_path; generated_at; run_id
@dataclass(frozen=True) class PageContext: meta; active; nav
def sparkline_svg(values, *, width=..., height=..., ...) -> str     # inline SVG, deterministic

class TrendQueries:
    def __init__(self, conn: sqlite3.Connection, rules: KeywordRules) -> None: ...
    def cloud(self, window, *, previous=None, limit=...) -> CloudDataset
    def latest_digests(self, *, limit=...) -> list[DigestSummary]
    # E4 extends this class with category-scoped digest queries + keyword-page datasets.
```

`TrendQueries` reads the **SQLite cache** directly (`open_cache(layout)`); it is
intentionally cache-aware and thrown away at the v2 boundary (PRD §15). Window
bounds come from an **injected** instant (`window_ending_at(instant, days=...)`);
`KeywordRules.apply(raw)` folds every raw keyword to its canonical form or
`None` if muted, applied at query time so alias/mute remaps are retroactive
(PRD §6). A keyword's count = its number of **distinct items** in the window.

### LLM provider (`grepify.llm`) - E2 (digest reuses it)

```python
class LlmClient:
    def complete(self, messages, *, run_id, purpose, input_items) -> Completion: ...
def build_client(profile, *, api_key, base_url, log_sink, clock, ...) -> LlmClient
```

Digest generation is `purpose='digest'` (PRD §6 `llm_log.purpose`). The budget
breaker refuses a call over `max_calls_per_run` with `BudgetExceededError`
**before** any network I/O; `LlmError` is raised when bounded retries are
exhausted. E4 catches both and degrades to a deterministic **template digest**
(PRD §13: "digest degrades to template") - the LLM failing never fails the run
or blocks the build (PRD §9). Secrets (`LLM_BASE_URL`/`LLM_API_KEY`) resolve
from the environment at wiring time, never from committed config (PRD §5).

### Storage + config

```python
Repository.add_digest(digest: Digest) -> None          # overwrite by digest_id (idempotent)
Repository.rebuild_cache() -> None                      # JSONL truth -> cache (digests included)
ConfigProvider.groups() -> list[SourceGroup]            # .category is the digest unit
```

`Digest` (`grepify.models`, PRD §6): `digest_id, kind ∈ {daily,weekly},
category, period_start, period_end, title, body_md, top_keywords (json
[{keyword,count}]), model, created_at`. Stored to `data/digests/<id>.json`.

## Interfaces / contracts produced

### Digest identity + url scheme

- `digest_id` = `daily-<category>-<YYYY-MM-DD>` | `weekly-<category>-<YYYY-Www>`
  (PRD §6). The date/week is the **just-completed** America/Edmonton period the
  digest covers (yesterday for daily, last ISO week for weekly).
- Page url = `/<base>/digest/<kind>/<slug>/` where `<slug>` = `digest_id`
  with its redundant `<kind>-` prefix removed (e.g. `/digest/daily/ai-2026-07-07/`).
  This matches trendcloud's `/digest/daily/<date>-<slug>` shape (a per-kind
  directory + a unique readable slug) and derives purely from `digest_id`
  (`kind` is a fixed-enum prefix, so stripping it is unambiguous even when the
  category itself contains a hyphen, e.g. `data-eng`). `digest_slug(digest_id,
  kind)` is the single helper the home list, digest index, and keyword-chip
  links all call.
- Keyword page url = `/<base>/keyword/<slug>/` where `<slug>` =
  `slugify(keyword) + "-" + blake2b(keyword)[:6]`. The hash suffix guarantees
  uniqueness + determinism for symbol-heavy or colliding keywords (`c++`, `c#`);
  `keyword_slug(keyword)` is the shared helper (home cloud + keyword index +
  digest chips).

### Rising detection (F-TRD-03, deterministic, config-driven)

A keyword is **rising** in a window iff `count >= rising_min_count` (default 3)
**and** the window-over-window ratio `count / previous_count >= rising_ratio`
(default 3.0); a keyword with `previous_count == 0` and `count >=
rising_min_count` is rising (surged from nothing). Pure function
`is_rising(count, previous_count, *, min_count, ratio)` - no clock, no I/O.
Feeds the digest prompt and a badge in the cloud. Thresholds live in
`settings.digest` (`rising_min_count`, `rising_ratio`).

### Digest input assembler (GRP-40, per category, deterministic)

`assemble_digest_input(queries, *, category, window, kind, settings) ->
DigestInput` gathers, for one category and window: the category's in-window item
count (skip-threshold input), its top-N keywords (distinct-item counts after
alias/mute, rising-flagged, with per-keyword top item titles/summaries), and the
subset flagged rising. Category scope = `items -> sources -> source_groups.category`
join. All lists sorted (`(-count, keyword)`, items by `published_at desc`) for
byte-stable output. **Never per-user** (PRD §2/§7: digests key on category).

### Digest generation (GRP-41/42)

`generate_digest(input, client, *, run_id, clock) -> Digest`. Below
`settings.digest.min_items` items in the category ⇒ **skipped** (returns `None`,
logged - not failed, F-DIG-03). Otherwise one `client.complete(purpose='digest')`
returning strict JSON `{title, tldr: [...], body_md}`; `body_md` is stored as
composed markdown (**TL;DR** bullet list + narrative). `top_keywords` chips come
from the deterministic assembler, not the model. On `LlmError`/`BudgetExceededError`
a deterministic template digest is produced instead (`model='template'`). Prompt
is versioned (`digest-v1`); `model` records provenance (F-DIG-04).

### Cron gating (GRP-45, pure, Edmonton/DST-aware)

`digest_gate(instant) -> DigestGate(daily: bool, weekly: bool)` converts the
injected instant to America/Edmonton local time: `daily` true in the morning
window (05:00-08:00 local, the slot the 13:00-UTC cron lands in across both DST
offsets), `weekly` true when `daily` and it is Monday. Pure, tested at DST
transition edges. Exposed as `grepify digest-gate` (prints `daily=/weekly=` for
`$GITHUB_OUTPUT` / `eval`), replacing the GRP-06 bash placeholder.

## Contracts & rules

- **Determinism (F-SIT-08 / S8)**: no `datetime.now()` in the render or assemble
  path - the clock is injected; all dict/set iteration sorted before templating;
  snapshot/offline tests render twice and assert byte-identical output. Period
  and gate math use `zoneinfo` (`tzdata` is a dependency so it is portable).
- **Failure modes**: LLM down/over budget ⇒ template digest (run continues,
  site builds). Too few items ⇒ digest skipped, logged. A bad cache / bad config
  propagates from the layers that own it (`RepositoryError`/`ConfigError`); E4
  adds no new process-failing path.
- **No new secrets in `validate`/PR-triggered workflows** (PRD §5, public repo):
  the digest step consumes `LLM_*` only in the scheduled `pipeline` job.
